# Delivery Output Handoff Prompt

## 1. Purpose

本 Prompt 用于基于 BOMCASE 当前 Knowledge 结构生成面向研发 / UI / 跨部门阅读的交付文档。

定位说明：

- 本 Prompt 属于 `Delivery Output Loop`；
- 本 Prompt 用于生成交付物，不用于更新知识库事实层；
- 本 Prompt 不能替代 `Knowledge Update Loop`；
- 如发现事实层缺口，必须中止输出并回到 Knowledge Update Loop。

## 2. Required Inputs

使用本 Prompt 前，必须提供或确认以下信息：

- 页面名称：
- 输出对象：研发 / UI / 研发+UI / 老板 / 跨部门
- 输出类型：完整型交接文档 / 变更型交接文档 / 阅读优化版
- 输出模式：增量 / 全量
- 目标版本：
- 本期范围：
- 是否已完成 Knowledge Update Loop：是 / 否

补充说明：

- 输出模式由人工指定；
- AI 不自行判断旧文档是否可作为底座；
- AI 不自行决定轻量或全量；
- 输出模式只决定交付文档组织方式；
- 输出模式不决定是否需要更新事实层；
- 若“输出模式”缺失，必须停止并要求人工补充，不得自行推断。
- 增量模式下，“本次变化”字段必须由人工填写；
- AI 不得自行推断本次变更范围；
- 若“本次变化”缺失或表述过于模糊，必须停止并要求人工补充，不得自行补全。

## 3. Required Reading Order

生成交付文档前，必须按以下顺序读取：

1. `projects/bomcase/pages/{page}.md`
2. `projects/bomcase/decisions.md`
3. `projects/bomcase/pending.md`
4. 必要时读取 `projects/bomcase/sources/`
5. 不以 `projects/bomcase/outputs/handoff/` 作为事实源

## 4. Preflight Gate

在生成交付文档前，必须先完成以下判断：

- page brief 是否覆盖当前页面有效口径；
- decisions 是否存在相关边界判断；
- pending 是否存在待确认项；
- 本次输入是否包含事实变化；
- 本次输入是否只是输出优化；
- 是否需要回到 Knowledge Update Loop。

### 4.1 必须停止 Delivery Output Loop 的情况

如果出现以下任一情况，必须停止 Delivery Output Loop，并回到 Knowledge Update Loop：

- 新增页面结构；
- 新增展示条件；
- 新增状态文案；
- 新增交互规则；
- 新增字段来源或计算口径；
- 新增验收口径；
- 新增待确认问题；
- 形成稳定判断或边界取舍。

### 4.2 可继续 Delivery Output Loop 的情况

如果仅涉及以下内容，可继续 Delivery Output Loop：

- 增加研发影响标记；
- 调整阅读顺序；
- 增加模块前一句话定位；
- 优化标题；
- 调整表达密度；
- 补充版本说明；
- 不改变产品规则、展示条件、状态文案、验收口径。

### 4.3 输出模式与 Loop 前置关系

- 输出模式只决定交付文档形态，不决定是否更新事实层；
- 是否需要更新事实层由“是否存在事实变化”决定；
- 若本次包含事实变化，必须先完成 Knowledge Update Loop；
- 即使输出模式为增量，也不能绕过 Knowledge Update Loop；
- 即使输出模式为全量，也不得在交付文档中临时补事实。

## 5. Output Optimization Rules

沉淀 v1.1 / v1.3 经验，输出优化规则如下：

- 文档顶部应包含“本次更新说明”；
- 对研发影响较高的模块，应增加“研发影响标记”；
- 对高误解、高新增、跨角色影响模块，应在模块标题下增加一句话定位；
- 版本说明必须区分：
  - 产品事实变化；
  - 输出 / 阅读优化；
- 如果本次只是阅读优化，必须明确：
  - 不改变产品规则；
  - 不改变展示条件；
  - 不改变状态文案；
  - 不改变验收口径；
- 待确认事项必须来自 `pending.md`，不得临时编造；
- 已确认边界必须来自 `decisions.md` 或 page brief；
- 不得把 pending 写成 confirmed。

### 5.1 交付正文语言边界（面向研发 / UI / 测试 / 跨部门）

- `Knowledge Update Loop` / `Delivery Output Loop` 等术语仅用于生成过程控制，不用于交付正文；
- 面向外部读者的正文不得输出以下内容：
  - 不是事实源；
  - 不反向覆盖；
  - 不归档；
  - 不进入 sources；
  - 不进入 Knowledge Update Loop；
  - 不进入 Delivery Output Loop；
  - 内部知识库文件路径；
  - prompt 测试状态；
  - generated_from_prompt；
  - source / pages / pending / decisions 的内部路径说明；
- 如确需标记草稿状态，只允许在 frontmatter 或文件顶部 1 行短注中说明；
- 内部治理说明不得占用正文首屏；
- 面向研发正文应使用产品语言 / 实现语言，不使用知识库治理语言。
- 交付正文所有章节（含“不覆盖范围”“验收口径”）不得出现内部知识库路径、内部文件路径或内部治理语言；
- 正文禁止出现以下类型表达：
  - `source`
  - `page brief`
  - `pages`
  - `pending`
  - `decisions`
  - `outputs`
  - `prompt`
  - `Knowledge Update Loop`
  - `Delivery Output Loop`
  - `事实层`
  - `归档说明`
  - `生成规则`
  - `测试状态`
- 如需表达边界或未覆盖范围，必须转译为产品 / 研发可理解语言，例如：
  - 不推荐：`source 未确认事实`
  - 推荐：`未经产品确认的新增口径`
  - 不推荐：`pending 中未确认的问题`
  - 推荐：`仍待产品确认的问题`
  - 不推荐：`page brief 未覆盖内容`
  - 推荐：`当前页面交接口径未覆盖的内容`

### 5.2 版本说明语言规则

- 版本说明只能包含：
  - 本次产品事实变化；
  - 阅读 / 表达优化；
  - 不变更范围；
  - 待确认影响；
- 禁止在正文版本说明中描述：
  - prompt 测试过程；
  - 知识库路径；
  - 内部归档状态；
  - 是否进入某个 Loop；
  - 是否为事实源；
  - 是否反向覆盖；
- 推荐写法示例：
  - `本次为阅读结构优化，不新增页面规则，不改变展示条件、按钮状态、资源计算和验收口径。`

### 5.3 研发影响标记最小实现口径

- 面向研发输出时，研发影响标记不能只列“关注点”；
- 每个高影响模块至少应包含：
  - 模块职责；
  - 已确认展示口径；
  - 关键状态 / 交互；
  - 研发影响侧：前端 / 后端 / 前后端；
  - 待确认阻塞点；
  - 验收关注点；
- 禁止仅写类似：
  - `保底模块：关注展示条件、进度来源`
- 推荐表达方向：
  - `保底模块：展示口径、职责边界、待确认点与验收关注点成组呈现。`

### 5.4 待确认项分层规则（按阻塞程度）

- 待确认项必须按影响程度分组：
  - A. 阻塞开发启动：
    - 数据来源；
    - 触发条件；
    - 字段口径；
    - 计算 / 刷新依赖；
    - 接口依赖；
  - B. 不阻塞开发启动，但影响验收：
    - 展示精度；
    - 文案；
    - 视觉样式；
    - 动效；
    - 最终表现细节；
- 若无法判断阻塞等级，标记为：
  - `需产品确认优先级`
- 不得默认全部阻塞，也不得默认全部不阻塞。

### 5.5 阅读优化版定位（轻量补充）

- 若输出对象是研发且文档用于执行，不应生成纯摘要；
- 至少应包含：首屏摘要 + 高影响模块的最小实现口径；
- 阅读优化版更适合评审预读、老板快速浏览或跨部门同步。

### 5.6 阅读方式说明位置规则

- 不建议在文末单独输出“本文使用说明”；
- 如需说明阅读方式，应合并到首屏摘要或“不变更范围声明”；
- 阅读方式说明最多 1-2 句话；
- 不得占用正文主体或作为独立末尾章节。
- 增量模式下，不单独输出“阅读顺序”“研发优先阅读顺序”“本文使用说明”等独立章节；
- 阅读引导应合并在“本次版本更新说明”前后，用 1 句话完成；
- “本次版本更新说明”中的“查看位置”列即主要阅读索引，不在正文额外创建阅读路径模块；
- 推荐表达：
  - `研发可优先按下表“查看位置”跳读本次变更章节；未列入本次更新说明的内容，按“不变更范围声明”处理。`

## 6. Recommended Handoff Structure

### 6.1 完整型交接文档

建议章节：

1. 文档版本说明
2. 本次更新说明
3. 研发影响标记
4. 页面目标
5. 页面结构
6. 组件与字段
7. 核心页面口径
8. 页面状态
9. 交互规则
10. 空状态 / 异常状态
11. 待确认项
12. 验收口径
13. 不覆盖范围

### 6.2 变更型交接文档

建议章节：

1. 本次变更背景
2. 本期范围
3. 继承规则
4. 本次变更点
5. 不变更范围
6. 研发影响
7. 待确认项
8. 验收口径
9. 归档说明

### 6.3 输出模式执行规则（人工指定）

AI 仅检查输入中的“输出模式”是否明确，不自行判断输出类型。

#### A. 当输出模式 = 增量

- 必须输出完整交接文档上下文；
- 必须在顶部输出“本次版本更新说明”；
- 版本更新说明必须包含：
  - 修改项；
  - 变化类型；
  - 查看位置；
  - 影响范围；
  - 是否需确认；
- “查看位置”必须写为“章节编号 + 章节标题”，且优先指向当前输出文档内章节，例如：
  - 可用：`第 6.2 节「能量溢出提示」`
  - 不可用：`能量溢出提示模块`
- 不建议：`上一版第 6.2 节`
- 如需说明继承关系，可写“沿用上一版口径”，但不得将旧文档作为唯一阅读入口；
- 必须输出“不变更范围声明”；
- 在“不变更范围声明”中统一说明：
  - 未特别标注 `【本次调整】` / `【本次新增】` / `【本次延续】` / `【待确认】` 的章节，均沿用上一版口径；
- 增量模式正文主体结构必须与全量模式保持一致；
- 增量模式不得另起一套“变更报告结构”替代页面交接正文；
- 增量模式应在全量结构基础上增加顶部变更导航，而不是替换正文结构；
- 增量与全量的结构差异仅限于：
  - 顶部增加“本次版本更新说明”；
  - 顶部增加“不变更范围声明”；
  - 本次变化涉及的正文章节增加 `【本次调整】` / `【本次新增】` / `【本次延续】` / `【待确认】` 标记；
  - 未涉及章节保留原正文结构，不新增额外说明章节，不逐节标注“未变更”；
- 正文保留完整页面交接结构；
- 本次变更涉及章节必须增加清晰标记，例如：
  - `【本次调整】`
  - `【本次新增】`
  - `【本次延续】`
  - `【待确认】`
- AI 只能基于人工输入的“本次变化”决定哪些章节使用以上标记；
- 未变更章节不得删除；
- 未变更章节默认不逐节标注；
- 仅当某个未变更模块存在高误解风险时，才允许在该章节标题或段落中轻量标注 `【未变更】`；
- 未提及内容默认沿用上一版口径；
- 增量模式目标是让研发优先阅读变更，同时在同一份文档中获得完整上下文；
- 增量模式不得输出成只有变更清单的摘要文档；
- 增量模式不单独新增“阅读顺序 / 研发优先阅读顺序 / 本文使用说明”等阅读路径章节；
- 增量模式不得默认新增以下一级章节：
  - `继承规则`
  - `本次变更点`
  - `归档说明`
  - `本文使用说明`
  - `阅读顺序`
  - `研发优先阅读顺序`
- 不得将“继承规则”作为默认一级章节；如需表达继承关系，应并入“不变更范围声明”或对应正文业务章节；
- 不得将“本次变更点”作为默认一级章节；本次变化以顶部“本次版本更新说明”表格为主索引，细节落入对应正文章节；
- 测试草稿状态或归档状态仅允许写在 frontmatter 或文件顶部短注；不得在正文新增“归档说明”章节；
- 正式交付文档正文也不输出“归档说明”章节；
- 增量模式下，“不覆盖范围”章节也必须使用产品 / 研发语言，不得出现内部知识库术语、内部文件路径或治理流程说明；
- 待确认项按当前文档结构完整保留，并对本次变更相关项进行重点标记；
- 若涉及研发高影响模块，必须给出最小实现口径；
- AI 不自行判断旧文档是否可作为底座。

#### B. 当输出模式 = 全量

- 顶部仍必须输出“本次版本更新说明”；
- 输出当前版本完整交接文档；
- 正文必须覆盖完整页面交接内容；
- 全量模式不是摘要型重述；
- 全量模式不是标准版、压缩版、关键点版等中间态；
- 全量模式用于重建当前版本完整交付底座；
- 全量模式必须达到或超过上一版正式交付文档的有效交付标准；
- 全量模式输出应能作为当前版本完整实现与验收依据；
- 全量模式不要求逐字复制上一版正式交付文档；
- 但不得降低上一版仍有效内容的交付颗粒度；
- 全量模式优化目标是“完整性不下降，阅读效率提升”；
- 不需要逐节标注“未变更”；
- 不要求研发依赖上一版；
- 至少包含：
  - 研发影响标记；
  - 页面目标；
  - 页面结构；
  - 组件字段；
  - 状态 / 交互规则；
  - 空状态 / 异常状态；
  - 待确认项；
  - 验收口径；
  - 不覆盖范围；
- 不得省略核心模块正文，仅输出摘要。
- 不得仅输出模块级、区域级、原则级摘要；
- 不得为了压缩篇幅丢失已确认或仍有效的字段、状态、交互、空状态、验收口径、待确认项。
- 上一版有效交付细节保留规则：
  - 当存在上一版正式交付文档，或已读取的 source material 中存在完整交付细节时，必须识别其中仍有效且未被 page brief / decisions / pending 否定的内容；
  - 全量模式必须保留这些有效交付内容的覆盖颗粒度；
  - 可重写表达、可合并重复、可调整章节顺序、可优化阅读路径、可增加研发影响标记；
  - 但不能丢失有效字段、状态、交互、验收、待确认项；
  - 不得将完整交付内容压缩为模块摘要或原则说明。
- 关键交付规则表定点保留规则：
  - 当上一版正式交付文档或已读取的 source material 中存在以下内容，且未被 page brief / decisions / pending 否定时，全量输出必须保留：
    - CTA 按钮状态表：应覆盖不同资源状态下的一键填充 / 开启军需按钮表现；
    - CTA 按钮状态表：应覆盖推荐态、灰度态、常规态、是否有特效；
    - CTA 按钮状态表：应覆盖条件不足时点击开启军需的处理边界；
    - CTA 按钮状态表：应覆盖保底模块、能量溢出提示是否影响按钮状态的边界；
    - info 入口说明内容：不仅说明存在 info 入口，还应保留对应说明文案；
    - info 入口说明内容：应覆盖保底 info、资源概览 info、能量余额 info 等已确认说明内容；
  - 允许优化表达，但不得仅写“hover 展示说明”而缺失具体说明内容。
- 组件与字段覆盖质量基准：
  - 不得只保留关键字段；
  - 不得将字段表压缩为区域说明；
  - 必须到字段粒度，不得只写区域级描述；
  - 应覆盖上一版仍有效的字段名、示例值、展示口径、状态文案、入口文案、与字段相关的边界说明；
  - 若上一版正式交付文档或已读取的 source material 中存在字段表，且未被 page brief / decisions / pending 否定，全量输出应保留字段表级别颗粒度；
  - 可以重组字段表，但不能把字段级内容压缩为模块级描述；
  - 不推荐仅写“左侧资源概览区包含若干资源字段”“右侧卡牌背包区展示卡牌相关信息”；
  - 推荐使用字段表或等价结构，至少包含字段名、示例值或口径说明。
- 状态覆盖质量基准：
  - 不得只保留关键状态；
  - 不得将状态表压缩为原则说明；
  - 必须按模块、场景展开，不得只保留原则性规则；
  - 应覆盖上一版仍有效的主要状态、触发条件、展示结果、不展示场景、状态切换后的联动影响、不适用处理；
  - 如上一版正式交付文档或已读取的 source material 中存在状态表、UI 状态规则或状态切换规则，且未被 page brief / decisions / pending 否定，全量输出应保留对应颗粒度；
  - 可以合并重复状态，但不能弱化状态边界；
  - 对关键切换（如 X1/X5）应明确清空哪些内容、重算哪些内容、哪些提示同步清空、哪些模块不受影响；
  - 应明确前端是否可自行推导，或必须以后端返回为准。
- 交互覆盖质量基准：
  - 不得只保留核心交互；
  - 不得将交互表压缩为动作列表；
  - 应覆盖上一版仍有效的触发对象、触发条件、结果表现、是否所有状态适用、不适用时如何处理；
  - 如上一版正式交付文档或已读取的 source material 中存在交互表，且未被 page brief / decisions / pending 否定，全量输出应保留对应颗粒度；
  - 可以调整交互章节结构，但不能丢失适用状态和不适用处理；
  - 对普通卡、已选卡、喜欢卡、清除已选、半展开入口、全展开入口等触发对象，应明确处理差异。
- 验收口径覆盖质量基准：
  - 验收口径必须可以直接转测试用例；
  - 不得只保留方向性验收；
  - 不得将详细验收清单压缩为原则性条目；
  - 应覆盖上一版仍有效的页面结构验收、字段展示验收、状态验收、交互验收、空状态验收、范围控制验收、与高影响模块相关的验收；
  - 应明确具体触发条件、具体展示结果、具体不展示场景、具体不得进入的区域；
  - 应明确待确认项不得作为已确认规则进行验收；
  - 如上一版正式交付文档或已读取的 source material 中存在详细验收口径，且未被 page brief / decisions / pending 否定，全量输出应保留对应颗粒度；
  - 可以合并重复验收项，但不能弱化验收条件或减少关键验收覆盖。
- 待确认项覆盖质量基准：
  - 不得遗漏上一版仍未关闭的待确认项；
  - 不得将待确认项写成已确认规则；
  - 不得减少问题数量或弱化影响说明；
  - 可以按阻塞程度重新分层，可以优化归属表达；
  - 但必须保留每个仍有效待确认项的核心问题、影响范围和确认责任方向。
- 不扩大复刻范围规则：
  - 本组补充规则不要求完整复制上一版正式交付文档；
  - 不要求视觉状态表数量、范围控制项数量、字段表边缘字段与上一版完全等量；
  - 其他细节可酌情取舍，取舍标准是是否影响研发实现或测试验收闭环；
  - 若会影响研发实现、联调、测试用例设计或验收判断，则必须保留；
  - 若仅属于视觉表达增强、历史说明冗余、已有功能内部细节，可合并或弱化。
- 全量结构保持规则：
  - 全量模式应保持清晰正文结构，不因补充关键规则表而退回冗长历史结构；
  - 推荐保持以下结构顺序：
    - 文档版本说明；
    - 本次更新说明；
    - 研发影响标记；
    - 按角色阅读路径；
    - 页面目标；
    - 页面结构；
    - 组件与字段；
    - 核心页面口径；
    - 页面状态；
    - 交互规则；
    - 空状态 / 异常状态；
    - 待确认项；
    - 验收口径；
    - 不覆盖范围；
  - 不得为追求细节等量而默认新增大量历史说明章节。
- 按角色阅读路径规则：
  - 全量模式下，“研发影响标记”后必须提供“按角色阅读路径”章节；
  - 按角色阅读路径只承担导航职责，不承担详细规则承载职责；
  - 角色阅读路径不得替代正文主体；
  - 角色阅读路径至少包含：前端、后端、UI / 设计、测试；
  - 角色路径条目只允许包含：章节编号、章节标题、该角色关注关键词、简短关注说明；
  - 角色路径中不得展开字段表、状态表、交互表、验收清单；
  - 前端阅读路径应指向组件与字段、核心页面口径、页面状态、交互规则、空状态 / 异常状态、验收口径等章节，并说明前端重点关注字段展示、状态联动、按钮状态、选卡交互、空状态与验收规则；
  - 后端阅读路径应指向研发影响标记、核心页面口径、待确认项、验收口径中与后端相关章节；
  - 后端阅读路径必须包含两个清单：
    - 后端需要确认的事项：应列出满足度与资源构成返回口径、保底配置与展示条件、保底进度/目标进度/剩余次数/保底奖励名称、开启后保底结算结果、X5 等批量开启最终保底结果、能量溢出值来源/刷新时机/展示精度/与最终余额沉淀一致性、及其他被标记为前后端或阻塞联调的问题；
    - 不作为后端改动依据的内容：应列出主要属于前端 / UI / 测试的展示细节，例如左侧信息层级、右侧半展开 / 全展开视觉表现、按钮推荐态的具体视觉表现、icon 具体样式、动效/颜色/描边/阴影/圆角/间距、info 弹层样式、卡牌选中态具体视觉实现等；
  - 后端确认范围必须使用产品 / 技术协作可理解语言，不得使用内部知识库治理语言；
  - 不得把前端 / UI 事项混写为后端必须处理事项；若某事项是否涉及后端不确定，应进入待确认项，不得写成已确认后端改动；
  - UI / 设计阅读路径应指向页面结构、组件与字段、页面状态、待确认项中的设计表达相关章节，重点关注产品表达约束、状态识别、视觉冲突避免、待确认视觉项；
  - 测试阅读路径应指向页面状态、交互规则、空状态 / 异常状态、待确认项、验收口径，重点关注状态覆盖、触发条件、期望结果、不展示场景、待确认项边界。
- 角色路径与正文职责分离规则：
  - 按角色阅读路径是正文前的导航层，不是规则承载层；
  - 正文主体仍必须保持全量交付颗粒度；
  - 不得因为已有角色路径而压缩组件与字段、核心页面口径、页面状态、交互规则、空状态 / 异常状态、待确认项、验收口径；
  - 角色路径可引用正文，但不能替代正文；
  - 全量正文密度应以 `open-box-handoff-full-unique-test-4.md` 的字段、状态、交互、CTA、info、验收闭环颗粒度为参考基线；
  - 新增角色路径后，正文密度不得低于上述基线。
- CTA 状态表表达规则：
  - CTA 状态表必须保留，但应以“表达约束 + 判定标准”为主，不写死具体视觉实现；
  - 推荐态判定标准至少包括：
    - 视觉权重显著高于非推荐态；
    - 用户不阅读按钮文案也能识别当前推荐操作；
    - 同一时刻仅有一个主要推荐操作；
  - 非推荐态判定标准至少包括：
    - 保持可识别、可点击；
    - 视觉权重弱于推荐态；
    - 不抢占推荐操作注意力；
  - 不可推荐 / 弱化态判定标准至少包括：
    - 在无可填充资源时，一键填充不作为推荐操作；
    - 开启军需在条件不足时仍保持可点击，并触发已有能力；
  - 不推荐将“强亮 + 呼吸光效”“灰度，无特效”“常规亮度”等具体视觉实现写成产品规则。
- 视觉表达边界规则：
  - 产品交接文档中的视觉相关内容应限定在“表达约束”层面；
  - 允许写：产品语义、展示条件、状态规则、可识别性要求、可区分性要求、必须避免的视觉冲突、不得误导用户的表达、不得抢占主操作权重的约束、以及视觉待确认项；
  - 不应写死：具体颜色、icon 具体造型、动效细节、描边粗细、阴影、圆角、间距、具体视觉风格、具体图标样式、具体动画曲线；
  - 不因补充 UI / 设计视角而写死具体视觉实现，不为对齐历史文档补回视觉细节等量内容。
- 产品规则 / 表达约束 / 设计实现判断标准：
  - 产品规则（必须写入产品交接文档）：例如喜欢卡需要保护标识、喜欢卡在一键填充时默认被保护、保底标记表示属于当前军需箱配置保底范围、未配置保底玩法时模块不展示不占位、X1/X5 切换后已选卡牌清空并触发状态重算；
  - 表达约束（应写入产品交接文档）：例如保护标识必须可识别且不被选中态覆盖、推荐操作按钮应显著于非推荐操作、保底标记应可与普通饰品区分、能量溢出提示为轻提示且不抢占中间主操作区、右侧半展开态不承载过重信息；
  - 表达约束应尽量包含可判断标准，不只写抽象要求；例如“可识别”“可区分”“不抢占主操作注意力”“不承载过重信息”等应可被测试验证；
  - 设计实现（不应由产品交接文档写死）：例如指定某一种 icon、固定未选/已选具体颜色、固定某一种光效、固定描边/阴影/圆角/间距/图标风格/动画曲线；
  - 具体视觉实现以最终 UI 设计稿或设计系统为准；UI 方案必须满足产品表达约束；产品交接文档不替代 UI 视觉规范；
  - 视觉细节待确认项可纳入 B 级待确认项，但不得写成已确认产品规则。
- 阅读优化边界：
  - 全量模式允许优化阅读，但阅读优化不得以压缩有效交付细节为代价；
  - 允许调整章节顺序、增加前置研发影响标记、增加版本更新说明、合并重复表达、将长段落转为表格、将分散口径归并到对应模块、用更清晰的标题和一句话定位提升阅读效率；
  - 禁止为了“更好读”删除字段级说明；
  - 禁止为了“更简洁”删除状态场景；
  - 禁止为了“更精炼”删除交互适用条件；
  - 禁止为了“减少篇幅”删除验收项；
  - 禁止只保留关键模块摘要。
- 与增量模式关系：
  - 本组规则只补充全量模式，不改变增量模式；
  - 增量模式适用于已有正式交付底座后的版本更新；
  - 全量模式适用于新页面首次交付、旧文档失效、或需要重建完整交付底座的场景；
  - 增量模式仍保持“全量正文结构 + 顶部变更说明 + 局部变更标记”；
  - 全量模式必须达到正式交付标准，不存在“标准版 / 摘要版 / 中间版”。

## 7. Output Boundary

边界要求：

- 交付文档是 outputs；
- 交付文档不是事实源；
- 生成结果应归档到 `projects/bomcase/outputs/handoff/`；
- 不得写入 `sources/`；
- 不得反向覆盖 page brief / decisions / pending。
- Loop 术语与知识库治理说明仅用于内部生成控制，不作为交付正文内容。
- 交付正文中的“验收口径”仅描述产品功能、页面行为、研发实现与测试验收；
- 不得在正文“验收口径”中写入文档生成质量、版本说明格式、章节索引格式或 prompt 执行情况；
- 上述生成质量与格式校验内容仅允许出现在生成报告或人工校验报告中，不进入交付正文。

## 8. Archive Rule

推荐归档方式：

- 文件路径：`projects/bomcase/outputs/handoff/YYYY-MM-page-name-handoff-vX.X.md`
- 如果只是当前执行版本，优先只归档最新版本；
- 不强制补齐所有历史版本。

归档文件建议包含 frontmatter 字段：

- `type: delivered_artifact_snapshot`
- `artifact_type: handoff_document`
- `status: archived`
- `version`
- `delivered_to`
- `archive_layer`
- `knowledge_synced`
- `note: 非事实源，不得反向覆盖知识层`

## 10. Validation Checklist

生成后必须校验：

- 是否基于最新 page brief；
- 是否引用 decisions；
- 是否保留 pending 为待确认；
- 是否已明确“输出模式”（增量 / 全量）；
- 是否没有新增事实；
- 是否没有把输出优化写入事实层；
- 是否明确版本变化类型；
- 是否可归档到 outputs/handoff；
- 是否未污染 sources。
